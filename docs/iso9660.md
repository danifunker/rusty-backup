# ISO 9660 Export — Implementation Plan

Add a true **ISO 9660** output option to the disk-image export feature, using the
[`hadris-iso`](https://crates.io/crates/hadris-iso) crate as the writer.

Unlike the `Raw`/`Vhd`/`2MG` exports, which just copy the source bytes (with or
without wrappers), an ISO export **re-encodes the logical file contents of the
source** into a fresh ISO 9660 filesystem. The user can then mount, burn, or
boot the resulting `.iso`.

This is substantially more work than the other export formats because the
exporter must walk the source filesystem, not just stream its bytes.

## Prior art / context

- `src/rbformats/export.rs` — `ExportFormat` enum + dispatch. All new work plugs
  in here.
- `src/fs/filesystem.rs` — `Filesystem` trait (`root`, `list_directory`,
  `read_file`, `write_file_to`, `write_resource_fork_to`, `volume_label`, …).
- Filesystem readers already implementing the trait: `fat`, `exfat`, `ntfs`,
  `ext`, `btrfs`, `hfs`, `hfsplus`, `prodos`. Source coverage is broad.
- `opticaldiscs-rs` is **read-only** — no writer lives there. The ISO writer
  will live inside `rusty-backup` (or a new sibling crate if we want reuse).

## Goal

Add `ExportFormat::Iso`, selectable from the same GUI entry points as the other
export formats (`gui/backup_tab.rs`, `gui/optical_tab.rs`, wherever else
`ExportFormat` is listed), producing a genuine, mountable ISO 9660 image with
Joliet + Rock Ridge extensions.

## Sessions

Each session is meant to be a single-sitting chunk of work that leaves `cargo
build` + `cargo test` green.

### Session 1 — Dependency + spike

**Goal:** confirm `hadris-iso` does what we think it does; get a tiny
end-to-end ISO produced from a hand-built file list.

- Add `hadris-iso` to `Cargo.toml` with the `std` feature.
- Write an **example** under `examples/iso_spike.rs` that:
  - Creates an in-memory tree of 2–3 files (e.g., `README.txt`, `dir/hello.bin`).
  - Builds an ISO with Joliet + Rock Ridge enabled.
  - Writes it to a tempfile.
- Verify by:
  - `file foo.iso` reports ISO 9660.
  - macOS `hdiutil attach foo.iso` mounts it cleanly.
  - `isoinfo -lR -i foo.iso` (if available) lists the files with long names.
- Document any surprises (API quirks, required traits, whether we can stream
  or must buffer) in this file under "Notes from Session 1".

**Exit criterion:** a working example and an updated notes section below.

### Session 2 — `FilesystemTreeWalker` abstraction

**Goal:** a helper that turns any `&mut dyn Filesystem` into something
`hadris-iso` can consume. This isolates the adapter from the writer, so Session
3 can plug it in without touching filesystem readers.

- New module `src/rbformats/iso_writer.rs`.
- Define a small internal type `IsoEntry { path: PathBuf, kind: IsoEntryKind }`
  where `IsoEntryKind` is `Dir` or `File { size: u64 }`.
- Implement a recursive walker: start at `fs.root()`, call `list_directory`,
  emit entries depth-first.
- For file contents, use `fs.write_file_to(entry, writer)` so we stream rather
  than slurping whole files into memory.
- Unit tests using the existing `MockFs` in `src/fs/tree.rs`.

**Exit criterion:** walker tests pass; the walker does not depend on
`hadris-iso` yet — that wiring happens in Session 3.

### Session 3 — `ExportFormat::Iso` plumbing

**Goal:** add the enum variant and hook it into the export dispatcher, even if
the actual writer path is a `todo!()` stub.

- In `src/rbformats/export.rs`:
  - Add `Iso` variant to `ExportFormat`.
  - `extension()` → `"iso"`.
  - `description()` → `"ISO 9660"`.
  - `dialog_filter()` → `("ISO 9660 Images", &["iso"])`.
  - `is_floppy_only()` → `false`.
  - In `export_whole_disk` / `export_partition` / `export_clonezilla_*`, add an
    `if format == ExportFormat::Iso { return export_iso_*(...); }` branch that
    for now returns `anyhow::bail!("ISO export not yet implemented")`.
  - Update the `test_export_format_*` unit tests.
- GUI: add the `Iso` variant to every `ExportFormat` picker / match so the
  option appears and the save dialog filters correctly.
- Confirm a build with `cargo build && cargo clippy -- -D warnings`.

**Exit criterion:** `Iso` appears in the UI, picking it produces a clear
"not yet implemented" error, nothing else regresses.

### Session 4 — Partition-level ISO export (the real one)

**Goal:** make `export_partition` with `ExportFormat::Iso` actually produce an
ISO from a single partition (or superfloppy) source.

- New function `src/rbformats/iso_writer.rs::write_iso_from_filesystem`:
  - Takes `&mut dyn Filesystem`, `dest_path: &Path`, progress/cancel/log
    callbacks, and an optional volume label.
  - Walks the filesystem (Session 2 walker).
  - Uses `hadris-iso` to build the ISO with Joliet + Rock Ridge.
  - Sets the volume ID from `fs.volume_label()` (fallback to stem of dest path,
    uppercased, ≤32 chars).
  - Calls `progress_cb` per file or per N bytes written.
- In `export.rs::export_partition`, when format is `Iso`:
  - Decompress the source to a tempfile (or `Cursor` if small enough).
  - Detect the filesystem type (reuse `fs::detect_filesystem` or equivalent).
  - Open the `Filesystem` impl.
  - Call `write_iso_from_filesystem`.
- **Start with FAT12/16/32** as the first supported source — floppies are the
  simplest case and the one explicitly on the roadmap.
- Test with:
  - A FAT12 floppy image (like the RTL8139 driver disk the user referenced).
  - A FAT32 partition dump.
  - Verify the resulting ISO mounts and file contents byte-match the source.

**Exit criterion:** a FAT source → ISO export round-trips file contents and
names correctly, verified on macOS mount.

### Session 5 — Additional source filesystems

**Goal:** expand supported source filesystems one at a time. Each is its own
mini-session if they turn out to be thorny.

Order of attack (easiest first):
1. **exFAT** — very similar to FAT.
2. **NTFS** — Unicode names, watch for ADS (alternate data streams — skip).
3. **ext2/3/4** — Unix perms via Rock Ridge; symlinks (see caveats).
4. **HFS / HFS+** — Mac metadata: type/creator codes, resource forks.
   Probable compromise: drop forks, or encode as AppleDouble `._` files.
5. **ProDOS** — Apple II; filename translation; skip for v1 if tricky.
6. **btrfs** — snapshots/subvolumes — scope to a single subvolume.

Gate each behind a feature-detection check so an unsupported source returns a
clear error rather than a garbled ISO.

### Session 6 — Whole-disk export strategy

**Goal:** decide and implement what `export_whole_disk` with `Iso` means.

An ISO 9660 image has **no partition table** — so "export a multi-partition
HDD as an ISO" is undefined. Options:

- **Option A (simplest):** error out with a helpful message; ISO export is
  partition-only. Document in UI.
- **Option B:** if the disk has exactly one filesystem partition, export it.
  Otherwise error.
- **Option C:** export the largest readable data partition with a warning.
- **Option D (ambitious):** create an ISO containing per-partition folders,
  each holding that partition's file tree. Not a 1:1 disk image but gives the
  user "all the files from the disk."

**Recommendation:** ship Option A first; revisit after user feedback.

### Session 7 — Tests, docs, polish

- Integration tests under `tests/` exercising FAT → ISO and ext → ISO
  round-trips.
- Update README / help text.
- Confirm progress bar + cancellation wire up correctly (may require buffering
  if `hadris-iso` doesn't expose per-file callbacks — see caveats).
- Run full `cargo fmt && cargo clippy -- -D warnings && cargo test`.

## Caveats / open questions (circle back on these)

**Metadata loss by design.** ISO 9660 + Joliet + Rock Ridge can carry Unix
perms, long Unicode names, symlinks, and timestamps — but they cannot carry:
- NTFS ACLs and alternate data streams.
- HFS/HFS+ Finder type/creator codes.
- HFS/HFS+ resource forks (workaround: emit AppleDouble `._filename` sidecars,
  which Mac Finder will re-fuse on read — confirm `hadris-iso` doesn't mangle
  dotfiles).
- ext4 xattrs beyond standard POSIX bits.
- File creation times (RR only records mtime/atime/ctime POSIX-style).

**Filename collisions.** Multiple files with names that differ only by case,
or names containing characters illegal in ISO / Joliet, may collide after
translation. Need a deterministic mangling scheme and a collision report.

**Large files.** ISO 9660 single-extent file size limit is 4 GiB − 1. Larger
files require multi-extent records — verify `hadris-iso` handles this, and if
not, either error out or fall back to UDF (not in scope for v1).

**Symlinks.** Only representable with Rock Ridge. Windows/macOS readers may
ignore them. Document as "best effort on Unix-style consumers."

**Empty directories.** Some ISO writers strip them; some don't. Pick a policy
and document.

**Sparse files.** Source filesystems may report a large logical size with
mostly zeros. ISO writes them literally — a 10 GB sparse file becomes a 10 GB
ISO entry. Consider a "materialize sparse runs as zeros" warning, or skip.

**Raw / non-filesystem bytes are lost.** Boot sectors, unallocated space,
inter-partition gaps, resource data outside the catalog — none of it survives.
If that's what the user wants, they should use `Raw` or `Vhd`, not `Iso`. The
UI tooltip should say this plainly.

**Bootability is not preserved.** A bootable floppy does not become a bootable
ISO for free — that requires El Torito boot-catalog setup with a boot image.
Out of scope for v1; revisit if users ask.

**Progress / cancellation granularity.** Depends on whether `hadris-iso`
exposes per-file or per-byte hooks during writing. If it only offers a
"finalize" call that writes the whole ISO at once, progress will jump from 0%
to "writing metadata" to 100%. Verify in Session 1 and note below.

**Memory usage.** If `hadris-iso` requires the full file list + directory
records in memory before writing, very large trees (millions of entries) could
be a problem. Floppies and CDs are fine; multi-TB HDDs are not. Measure in
Session 4 and cap tree size if needed.

**Licensing.** `hadris-iso` is dual-licensed MIT OR GPL-3.0. Ensure
`rusty-backup`'s license is compatible with the MIT arm (it almost certainly
is); otherwise we must take the GPL arm and audit the distribution impact.

**Whole-disk ambiguity.** See Session 6. Decision still deferred.

**Filesystem coverage.** Source filesystems rusty-backup cannot read today
(e.g., ReFS, ZFS) obviously can't be ISO-exported either. Error messages must
be specific.

**Unicode normalization.** HFS+ stores NFD, ext stores opaque bytes, NTFS
stores UTF-16. Joliet wants UCS-2. Pick an NFC normalization pass at export
time and document it.
